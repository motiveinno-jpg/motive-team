# [S4-미국바이어] buyer-app 내부 네비게이션 전면 깨짐 — 아키텍처 결함 | P0 | buyer-app.html

## 요약
buyer-app.html에서 대시보드 이외의 **어떤 내부 페이지로든 이동하면 제조사 앱(whistle-app.htm)으로 전환됨**.
이것은 S4-008, S4-009의 상위 버그이며, buyer-app의 SPA 라우팅이 근본적으로 깨져있음.

## 재현 경로
1. /app/buyer에서 바이어 로그인 → 대시보드 정상 표시
2. 다음 중 하나 시도:
   a) 사이드바 "Messages" 클릭 → /app (제조사 앱, 일본어)
   b) 사이드바 "Orders & Shipping" 클릭 → /app (제조사 앱, Yamamoto 계정)
   c) "Sign Up" 탭 클릭 → /app (제조사 가입 폼)
   d) B.set({page:'search'}) 호출 → /app (제조사 앱)
   e) B.nav('search') 호출 → /app (제조사 앱)
   f) 뷰포트 변경 + 리로드 → /app (제조사 앱)
   g) 페이지 새로고침 → /app (제조사 앱)

## 영향
- **바이어 앱의 대시보드만 사용 가능**, 나머지 전체 기능 접근 불가
- 제품 검색, 채팅, 주문, 서류, 설정 등 모든 서브페이지 접근 불가
- 바이어 신규 가입 불가
- **바이어 앱이 사실상 데모 수준 — 프로덕션 사용 불가**

## 근본 원인 분석
1. buyer-app.htm과 whistle-app.htm이 **같은 도메인**에서 서비스됨
2. Supabase auth 세션이 localStorage에 **도메인 단위**로 저장 → 양 앱이 동일 세션 공유
3. buyer-app의 내부 이벤트(onclick, B.set, hash 변경)가 Cloudflare Pages의 _redirects 규칙을 트리거:
   - buyer-app 내에서 페이지 변경 → URL 경로 변경 → /app/* 규칙이 whistle-app.htm을 서빙
4. buyer-app의 B 객체와 whistle-app의 B 객체가 같은 전역 스코프에서 충돌

## 제안 해결방향
1. **단기**: buyer-app의 모든 내부 네비게이션을 hash 기반 (#search, #messages)으로 통일하고 URL path 변경 금지
2. **중기**: buyer-app을 /buyer/* 경로로 분리하여 /app/* 와 충돌 방지
3. **장기**: buyer-app과 whistle-app의 Supabase 세션 관리 분리 (다른 storage key 또는 subdomain)
